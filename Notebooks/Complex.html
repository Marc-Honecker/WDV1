Komplexe Zahlen und Rechenregeln mit Gleitkomma-Zahlen#

Im ersten Teil diesen Kapitels werden wir uns um das Rechenen mit komplexen Zahlen mit Python kümmern. Im zweiten Teil von diesem Kapitel schauen wir uns dann noch Probleme an, die beim Rechnen mit Gleitkomma-Zahlen entstehen.

Komplexe Zahlen#

Schauen wir uns am besten zuerst an, wie man eine komplexe Zahl in Python erstellt:

complex_num = 2.5 + 4j

complex_num

(2.5+4j)

Das erste, was sehr stark ins Auge fällt, ist mit Sicherheit, dass das Symbol der imaginären Einheit i in Python durch ein j dargestellt wird. Die Gründe hierfür ist total praktischer Natur:

  • Der Buchstabe i wird oft als Laufindex in einer Schleife genutzt und dies würde dann zu Konfusion führen.

  • Zusätzlich kann man i sehr leicht mit dem Buchstaben l oder 1 verwechseln.

Außerdem fällt auf, dass wir komplexe Zahlen ohne irgenwelche imports benutzen können. Rechnen kann man mit komplexen Zahlen, wie man es erwarten würde:

a = 2.5 + 3j
b = 1 + 2.5j

# Addition
print(a + b)
# Subtraktion
print(a - b)
# Multiplikation
print(a * b)
# Division
print(a / b)

(3.5+5.5j)
(1.5+0.5j)
(-5+9.25j)
(1.3793103448275863-0.44827586206896547j)

Ebenso kann man auch Listen von komplexen Zahlen erstellen:

complex_ls = list()

for num in range(10):
    num = num + 10j
    complex_ls.append(num)

complex_ls

[10j,
 (1+10j),
 (2+10j),
 (3+10j),
 (4+10j),
 (5+10j),
 (6+10j),
 (7+10j),
 (8+10j),
 (9+10j)]

Rechenregeln mit Gleitkommazahlen#

Wie der ein oder andere im Laufe des Kurses gemerkt haben könnte, können beim Rechnen mit Gleitkommazahlen einige Probleme entstehen. Schauen wir uns den folgenden Code einmal an:

a = 5.001
b = 2.001 + 3.0

a == b

False

Offensichtlich sollte 5.001 == 2.001 + 3.0 gelten, aber Python sagt uns, dass dies falsch ist! Jedoch warum ist dies falsch? Dies liegt ganz einfach daran, dass wir Menschen im Zehner-System rechnen, aber ein Computer im Binär-System rechnet mit endlicher Genauigkeit. Und bei dieser Umwandlung kommt es zu solch skurilen Problemen (die an einigen Stellen zu großen Problemen geführt haben). Deshalb darf man niemals Gleitkomma-Zahlen direkt auf Gleichheit prüfen.

Sicherer ist es, Gleitkomma-Zahlen auf Ungleichheit zu prüfen. Ein Beispiel:

a = 2.000001
b = 2.0

a > b

True

Obwohl die Zahlen, die wir hier gewählt haben, sehr viel dichter beieinander liegen, als in dem Beispiel zuvor, verhält sich der Vergleich genauso, wie man es vermuten würde. Deshalb sollte man auch bei einem Test auf Gleichheit zweier Gleitkomma- Zahlen einen sogenannten Epsilon-Test durchführen. Dieses Konzept ist allerdings außerhalb der Abdeckung dieser Vorlesung.
